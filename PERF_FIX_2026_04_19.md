## Fix 1 — CrystallizationBuffer vectorised add()

**Symptom:** Training throughput degraded from ~1 s/it to ~11 s/it exactly when
`train/crystal/buffer_fill` saturated at 1.0 (step ~42). Observed in W&B run
`vanilla-skink`.

**Root cause:** The old `add()` method iterated over each element of the batch
in a Python loop (`for i in range(B)`), doing 768 individual tensor operations
per step (B=384 batches × 2 tensors). At saturation, list-index assignment
(`self.keys[pointer] = ...`) became slower than append due to Python list overhead
and per-element tensor operations.

**Fix:** Replaced list-of-small-tensors with two pre-allocated CPU tensors
`[capacity, key_dim]` and `[capacity, value_dim]`. The `add()` method now:
1. Does a single GPU→CPU transfer per tensor (not per element).
2. Computes ring-buffer destination indices with `torch.arange % capacity`.
3. Writes via a single vectorised slice assignment.

No Python loop per `add()` call. Post-saturation throughput is within 1.5× of
pre-saturation throughput (verified by `test_buffer_add_throughput` on CUDA).

**API preserved:** `add(keys, values)`, `consolidate(...)`, `clear()`, `__len__()`
all have unchanged signatures. `consolidate()` now slices `self.keys[:self.size]`
instead of `torch.stack(self.keys)`.

## Fix 2 — torch version pin + flash-attn optional

**Symptom:** `pip install -r requirements.txt` on a fresh Vast.ai instance pulled
in torch 2.11 (latest at provisioning time). No prebuilt flash-attn wheels exist
for torch 2.11; source build took 20-40 minutes and failed with CUDA kernel errors.

**Fix:**
- Pinned torch to `>=2.4,<2.7` — all three versions have stable prebuilt flash-attn
  2.7.x wheels. PyTorch SDPA fallback in `layers.py` already handles the no-flash-attn
  case with negligible throughput cost on A100.
- Moved flash-attn to `[project.optional-dependencies]` in `pyproject.toml` with
  install instructions in `requirements.txt`.
- Added `Dockerfile` based on `pytorch/pytorch:2.6.0-cuda12.4-cudnn9-devel` with
  flash-attn prebuilt for sm_80 (A100). Using this image reduces provision time to ~2 min.

**Assumption:** The training code runs correctly on torch 2.4–2.6. No torch API used
in the codebase was introduced after 2.4 (confirmed by review of `layers.py`,
`coral_v3.py`, `act.py`).

## Phase 3a' Launch Instructions

```bash
# 1. Pull the updated repo on your GPU instance
git pull

# 2. Install / upgrade deps (torch pin is now >=2.4,<2.7)
pip install -e .

# 3. Launch Phase 3a' with the warm-start config
python scripts/train.py --config-name phase3a_crystal_warmstart

# 4. Monitor in W&B:
#    - train/crystal/buffer_fill should saturate at step ~26 WITHOUT throughput drop
#    - train/crystal/bypass_rate should exceed 0.20 by step ~10000
#    - train/crystal/codebook_utilisation should exceed 0.30 by step ~5000
```

Success criteria: `exact_accuracy >= 0.58`, `bypass_rate >= 0.20`, `codebook_utilisation >= 0.30`.
